# Архитектура проекта

Архитектура проекта спроектирована на стыке нескольких парадигм: **ECS (Entity Component System)** для игровой логики, **MVVM** для обработки ввода пользователя и **Unidirectional Data Flow** для UI. Такое разделение предотвращает появление "God Object" и спагетти-кода.

## Высокоуровневая структура

Приложение разбито на несколько обособленных слоев:

### 1. Слой приложения и цикл (Core Layer)
Главной точкой входа игровой части является класс `GameApp`. Он отвечает за инициализацию всех сервисов (рендерер, физика) и управление `GameLoop`. 
- `GameLoop` жестко привязан к `requestAnimationFrame` или фиксированным временным шагам (fixed time step), что критично для стабильного просчета физики и детерминированности.

### 2. Слой логики и данных (ECS Layer)
Основа геймплея — библиотека `bitECS`. 
- **Данные** лежат в строго типизированных массивах (Компонентах).
- **Сущности (Entities)** — это просто целочисленные ID, которые связывают компоненты.
- **Поведение** описывается Системами (`AISystem`, `WeaponSystem` и т.д.). Системы не имеют состояния, они просто итерируются по массивам, изменяя значения каждый кадр.

### 3. Слой сервисов (Service Layer)
Внешние движки и тяжеловесные зависимости обернуты в сервисные классы:
- `PhysicsEngine`: Инкапсулирует `matter-js`, скрывая его сложный API за понятными методами для текущего проекта.
- `PixiRenderer`: Обертка над `PIXI.Application`. Вся сцена, стейджи, камеры и ассеты хранятся здесь.
- `ObjectPool`: Управляет жизненным циклом сложных объектов, снижая нагрузку на Garbage Collector Chrome/Firefox/Safari.

### 4. Связующий слой и Состояние (State & Bridging Layer)
Для того чтобы React мог реагировать на изменения внутри игрового движка (например, изменение количества здоровья или подбор предмета), используется реактивный стейт `Zustand` (`GameStore.ts`). 
- Игровой движок (через системы) обновляет Store.
- React-компоненты "слушают" Store и мгновенно перерисовывают HUD.
- *Важно: Использовать Zustand для сохранения позиции объектов (X/Y) запрещено из-за низкой скорости обновления React по сравнению с 60+ FPS Canvas. Для координат используется напрямую ECS и Pixi.*
- **GameContext:** Передает текущее состояние игры (меню, уровень, смерть) для синхронизации логики систем.

### 5. Слой представления (Presentation Layer)
- **UIOverlay (React):** Верхний слой поверх Canvas. Рисует меню, инвентарь, шкалу здоровья. Использует React, Tailwind CSS и Motion (для анимаций UI). 
- **MainMenu (React):** Компонент главного меню, выступающий точкой входа пользователя.
- **Localization System:** Папка `src/localization/` хранит текстовые ресурсы, обеспечивая удобное управление языками и их использование во всем приложении (UI и игровые эффекты).
- **PixiRenderer / SpriteBuilder:** Нижний слой. Рендерит все игровые объекты, тайлсеты, эффекты частиц в WebGL-контекст.

## Поток управления (Сontrol Flow)

1. **Ввод данных (`InputSystem` + `InputViewModel`):** Пользователь нажимает кнопки/мышь. Ввод фильтруется через ViewModel и записывается в ECS (на сущность "Игрок").
2. **Просчет логики (Системы):** 
   - `AISystem` определяет путь врагов, обновляет их векторы.
   - `SpawnSystem` добавляет новые сущности (мобы/снаряды) при необходимости.
   - `WeaponSystem` рассчитывает стрельбу и кулдауны.
3. **Физика (`PhysicsEngine` + `PhysicsSyncSystem`):** `matter.js` просчитывает шаг. Затем ECS синхронизирует новые координаты из физического мира к компонентам позиций. Решаются коллизии (`CollisionSystem`).
4. **Рендеринг (`RenderSystem`):** Измененные координаты передаются спрайтам `Pixi.js`. Происходит отрисовка кадра на экране.

## Событийно-ориентированная часть (`EventBus`)
Для разорванных и редких событий (например, "Босс убит" или "Уровень завершен") используется классический шаблон Наблюдатель через кастомный `EventBus`. Это позволяет избежать постоянной проверки флагов в `GameLoop`.
